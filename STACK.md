# Video RAG — Технологический стек

## Обзор системы

Video RAG — система семантического поиска по видеоконтенту для онлайн-кинотеатра. Пользователь описывает сцену на естественном языке — система находит точный момент в фильме с таймкодом.

---

## Архитектура

Система состоит из двух контуров:

**Контур индексации** (offline, на GPU-кластере) — обрабатывает видео, извлекает смысл из каждой сцены и сохраняет в векторную базу.

**Контур поиска** (online, real-time) — принимает запрос пользователя, находит релевантные сцены через гибридный поиск, переранжирует результаты и генерирует ответ.

```
Видео → Сегментация → Транскрипция речи ─┐
                    → Описание кадров ─────┤
                    → OCR текст на экране ─┤
                                           ▼
                              Контекстное обогащение
                                           │
                                     Эмбеддинги
                                           │
                                      Qdrant DB
                                           │
               Запрос → Query Router → Hybrid Search → Реранкинг → LLM ответ → UI
```

## Компоненты

### 1. Сегментация видео на сцены

**PySceneDetect** — open-source библиотека для обнаружения монтажных склеек.

Анализирует цветовые гистограммы между соседними кадрами. Когда разница превышает порог — фиксирует границу сцены. Это даёт натуральное разбиение фильма на семантически целостные фрагменты (200–400 сцен для полнометражного фильма), в отличие от механической нарезки по фиксированным интервалам.

- Алгоритм: ContentDetector (HSV histogram diff)
- Лицензия: BSD-3
- GPU: не требуется
- Зависимости: OpenCV

---

### 2. Распознавание речи

**GigaAM-v3** — foundational ASR-модель от Сбера, обученная на 700 000 часов русской речи.

Архитектура Conformer с self-supervised pre-training (HuBERT-CTC). Версия 3 добавляет автоматическую расстановку пунктуации и капитализации. На русском языке превосходит Whisper large-v3 в 2–3 раза по метрике WER.

- Параметры: 240M
- Размер модели: 449 MB
- WER на русском: ~2–3% (Whisper large-v3: ~6–10%)
- Может работать на CPU быстрее, чем Whisper на GPU
- Не галлюцинирует на длинных аудиосегментах
- Лицензия: MIT
- Разработчик: SberDevices (Россия)

Варианты декодера:
- CTC — быстрее, чуть менее точный
- RNNT — точнее, чуть медленнее

---

### 3. Визуальное описание сцен

**Qwen3-VL-4B-Instruct** — мультимодальная vision-language модель от Alibaba.

Принимает на вход изображения и текст, генерирует текстовое описание. Для каждой сцены подаём 3–5 ключевых кадров вместе с транскриптом речи. Модель описывает: что происходит, кто присутствует, эмоции, обстановку, визуальные детали.

- Параметры: 4.4B
- VRAM: ~3–4 GB (INT4), ~10–12 GB (BF16)
- Поколение: Qwen3 (октябрь 2025) — новее и умнее Qwen2.5-VL-7B при вдвое меньшем размере
- Нативная обработка видео и изображений
- Контекстное окно: 256K токенов
- Поддержка 29 языков, включая русский
- Лицензия: Apache 2.0

Альтернатива при наличии запаса GPU — **MiniCPM-V 4.5** (8B): превосходит GPT-4o на бенчмарке OpenCompass, имеет 96-кратное сжатие видеотокенов (обрабатывает видео за 8.7% времени конкурентов).

---

### 4. Контекстное обогащение (Context Enrichment)

**Qwen3-14B** — текстовая LLM для генерации сюжетного контекста.

Двухэтапный процесс, основанный на методике Anthropic Contextual Retrieval (+49% к качеству поиска):

**Этап 1** — модель получает описания всех сцен фильма и генерирует краткое содержание: жанр, главные персонажи, сюжетные линии, ключевые повороты.

**Этап 2** — для каждой сцены модель генерирует контекстный префикс (2–3 предложения), объясняющий место сцены в общем сюжете с учётом соседних сцен.

Этот префикс встраивается в текст чанка перед эмбеддингом. Благодаря этому сцена, где героиня молча стоит у окна, содержит в тексте информацию "это момент когда она осознаёт предательство и решает уехать" — и находится по соответствующему запросу.

- Параметры: 14B
- VRAM: ~8–10 GB (INT4)
- Лицензия: Apache 2.0

---

### 5. Эмбеддинги

**BGE-M3** (BAAI) — модель текстовых эмбеддингов с тройной функциональностью.

Генерирует одновременно три типа представлений из одного текста:
- **Dense vectors** (1024 измерения) — семантический смысл
- **Sparse vectors** (лексические веса) — ключевые слова, аналог BM25
- **ColBERT vectors** — fine-grained token-level matching (опционально)

Это позволяет делать гибридный поиск в одном запросе: dense ловит семантику ("грустная сцена прощания"), sparse ловит точные слова ("Максим, прости меня").

- Размерность: 1024
- Поддержка: 100+ языков
- SOTA на бенчмарке MIRACL (multilingual retrieval)
- VRAM: ~2 GB
- Лицензия: MIT

---

### 6. Векторная база данных

**Qdrant** — векторная база данных с поддержкой гибридного поиска.

Хранит для каждого чанка: dense вектор (1024d), sparse вектор (BM25 токены) и payload с метаданными (ID фильма, номер сцены, таймкоды, тип чанка, текст).

Поиск:
- **Dense prefetch** — top-50 по косинусному сходству
- **Sparse prefetch** — top-50 по BM25 лексическому совпадению
- **RRF fusion** (Reciprocal Rank Fusion) — объединение двух списков в единый top-20
- **Metadata filtering** — фильтрация по ID фильма, типу чанка, временному диапазону

Для хакатона используется in-memory режим (данные в оперативной памяти). Для production — persistent storage с HNSW индексом.

- Язык: Rust
- Протокол: gRPC / REST
- Лицензия: Apache 2.0

---

### 7. Реранкинг

**FlashRank** — ультралёгкий cross-encoder реранкер на базе ONNX.

Выбран вместо bge-reranker-v2-m3 из-за требования latency < 1 секунда. FlashRank работает на CPU через ONNX Runtime и укладывается в ~100-200ms на 20 кандидатов, что оставляет бюджет для остальных компонентов.

В отличие от bi-encoder (BGE-M3), который кодирует запрос и документ независимо, cross-encoder обрабатывает пару (запрос, документ) совместно, что даёт более точную оценку релевантности. Применяется только к top-20 кандидатам и сокращает их до top-5.

- Работает на CPU (ONNX), не требует GPU
- Latency: ~100-200ms на 20 кандидатов
- Мультиязычный
- Лицензия: Apache 2.0

Альтернатива для offline-оценки: **bge-reranker-v2-m3** (тяжелее, но точнее).

---

### 8. Генерация ответа

**Qwen3-14B** (тот же экземпляр, что для enrichment) — генерирует финальный ответ пользователю.

Получает запрос + top-5 сцен с полным текстом, определяет наиболее подходящую сцену и объясняет почему она соответствует запросу. Возвращает структурированный JSON с индексом лучшей сцены и текстовым объяснением.

---

### 9. Пользовательский интерфейс

**Streamlit** — фреймворк для быстрого создания веб-интерфейсов на Python.

Интерфейс включает:
- Текстовое поле для поискового запроса
- Sidebar с выбором фильма (или "искать по всем")
- Блок ответа ИИ с объяснением
- Список найденных сцен с таймкодами и score релевантности
- Видеоплеер с автоматическим переходом на нужный таймкод и остановкой на конце сцены (custom HTML5 player)
- Превью ключевого кадра сцены

---

### 10. OCR текста на экране

**EasyOCR** — распознавание текста на ключевых кадрах (вывески, письма, титры, надписи на экране).

Добавляет третий сигнал к каждой сцене. Если в кадре есть текст — он извлекается и индексируется вместе с транскриптом и визуальным описанием. Это позволяет находить сцены по надписям, которые не были озвучены.

Пример: запрос "сцена с письмом" — OCR нашёл текст письма на экране, даже если персонаж его не зачитывал.

- 3 строки кода для интеграции
- Поддержка 80+ языков включая русский
- Работает на CPU
- Лицензия: Apache 2.0

Источник идеи: паттерн "auxiliary texts" из Video-RAG (NeurIPS 2025) — визуальные сигналы преобразуются в текст, который хорошо индексируется и "понимается" LLM.

---

### 11. Маршрутизация запросов (Query Router)

**LLM-классификатор** — определяет тип запроса и оптимизирует стратегию поиска.

Перед поиском LLM анализирует запрос пользователя и определяет его тип:
- **Диалоговый** ("где герой говорит 'я знал с самого начала'") → приоритет транскриптам, BM25 весит больше
- **Визуальный** ("сцена под дождём на мосту") → приоритет VLM-описаниям
- **Сюжетный** ("почему герой решил уехать") → приоритет контекстным префиксам, расширенный контекст

Реализуется как один вызов LLM с structured output. Занимает ~2 часа на реализацию, но выглядит как "агентная архитектура" на демо — сильное впечатление на жюри.

---

### 12. Оценка качества (Kaggle метрики)

Соревнование оценивает по двум метрикам:

- **SuccessRate@K (SR@K)** — среди top-K есть предсказание с правильным видео И IoU ≥ 0.5 с ground truth сегментом
- **VideoRecall@K (VR@K)** — среди top-K есть правильное видео (без учёта таймкодов)
- **FinalScore = (AvgSR + AvgVR) / 2**, где Avg = среднее по K=1,3,5

50% скора — найти правильное видео. 50% — найти правильный момент.

Стратегии для максимизации:
- **Sentence-level chunking** с точными таймкодами → высокий IoU для SR@K
- **Video-level summary чанки** → гарантированный VR@K
- **Window padding ±3-5 сек** → повышает IoU с ground truth
- **Ablation study**: transcript-only → +VLM → +enrichment → +reranker

## Обоснование архитектурных решений

### Почему Text Grounding, а не нативные мультимодальные эмбеддинги

Доминирующий паттерн 2025 года — конвертация всех модальностей видео в текст (через ASR, VLM-описания, OCR), затем стандартный текстовый RAG. Это подтверждено:

- **Ragie** (production-система): VLM-описания дают лучшее качество retrieval при **2x меньшей латентности и 6x меньшей стоимости** по сравнению с нативными мультимодальными эмбеддингами
- **NVIDIA Developer Blog**: рекомендует Text Grounding как основной паттерн для Video RAG
- **Video-RAG** (NeurIPS 2025): "auxiliary texts" (ASR + OCR + object tags) как plug-and-play усиление любой VLM

Причина: текстовые эмбеддинги более зрелые и имеют лучшее семантическое покрытие, чем CLIP-подобные joint embeddings.

### Почему GigaAM-v3, а не Whisper

Оба PDF-исследования рекомендуют Whisper/faster-whisper. Мы сознательно заменяем на GigaAM-v3 по трём причинам:

1. **Качество на русском**: WER ~2–3% vs ~6–10% у Whisper (в 2–3 раза лучше)
2. **Отсутствие галлюцинаций**: Whisper вставляет ложные фразы на тишине/музыке — это портит индекс и retrieval. GigaAM стабилен
3. **Доступность в РФ**: полностью российская разработка, MIT лицензия, нет зависимости от OpenAI

### Почему Contextual Enrichment, а не Knowledge Graph

Оба исследования упоминают Graph RAG / SceneRAG как перспективный подход. Мы выбираем Contextual Enrichment потому что:

- Даёт **80% эффекта Graph RAG** при **5% сложности реализации**
- Реализуется за 3–4 часа (vs недели для полноценного KG)
- По методике Anthropic: **+49% к качеству retrieval**
- Не требует NER, face recognition, entity resolution

### Почему Evaluation Set — обязательная часть

Оба PDF-исследования настоятельно рекомендуют минимальную оценку. Жюри хакатона оценивает:

1. **Recall@1** — "первый результат правильный" — это то, что видно в демо
2. **Латентность** — <3 секунды для интерактивного демо
3. **Wow-фактор** — 5–10 впечатляющих запросов
4. **Ablation** — показать что каждый компонент добавляет ценность

20–30 размеченных запросов достаточно для убедительной оценки без большого датасета.

## Что мы сознательно НЕ строим

| Не делаем | Почему |
|-----------|--------|
| Полный Knowledge Graph | Недели работы, высокий риск не довести до демо |
| Face recognition pipeline | Технически сложно, юридически рискованно |
| 72B модели локально | Медленно, риск OOM, нет заметного выигрыша для описаний сцен |
| Temporal grounding с нуля | Нет готовых весов, рискованно по данным/времени |
| >10 часов видеоконтента | Лучше качественно обработать 2–3 часа |
| Streaming video processing | Слишком сложно для хакатона |

## Позиционирование для жюри

**Наш нарратив**: "Мы построили мультимодальную систему, которая понимает видео на уровне сцен, комбинирует визуальный и диалоговый анализ, и возвращает точный момент с объяснением."

**Демо-сценарий** (5 минут):
1. Базовый запрос по диалогу → система находит точный момент
2. Визуальный запрос ("сцены с дождём") → VLM-описания работают
3. Сюжетный запрос ("почему герой решил X") → contextual enrichment в действии
4. Показать Query Router: система различает типы запросов
5. Показать метрики: "мы протестировали на 30 запросах, Recall@3 = X%"
6. Показать ablation: текст-only → +визуал → +enrichment

**Техническая глубина**: объяснить hybrid retrieval (BM25 + dense + RRF), показать разницу между text-only и multimodal результатами, продемонстрировать reranking.
